# Принципы корректности кода (границы с внешним миром)

> Каждый принцип — вывод из конкретного латентного бага, найденного при проверке
> `# verify`-мест против реально установленных библиотек. Рядом — требование.

---

### 1. Нормализуй внешние идентификаторы к канонической форме на входе
Значения из внешних источников (библиотека, конфиг, сеть, пользователь) различаются
регистром, кодировкой, пробелами. Нельзя сравнивать «сырые» значения из разных
источников на равенство.

**Случай:** библиотека отдавала `pubkey_prefix` через `bytes.hex()` (всегда нижний
регистр), а тот же ключ из конфига не нормализовался. Регистрозависимое сравнение
не совпадало → **тихий дроп всех входящих сообщений** без ошибки в логе.

**Требование:** нормализуй внешние идентификаторы (регистр/кодировку/формат) в
**одной** точке на входе (валидатор схемы, парсер) — дальше по коду они уже
канонические. Сравнивай канон с каноном, а не сырое с сырым.

### 2. Проверяй ограничения на самой ранней и дешёвой границе
Структурные инварианты (длина, формат, диапазон) должны проверяться там, где ошибка
дешёвая и понятная — на уровне конфига/схемы, а не там, где значение наконец
«стрельнёт» глубоко в IO или на железе.

**Случай:** PSK private-канала не валидировался по длине. Кривой ключ всплывал бы
только при записи канала на устройство — как загадочный `ValueError` из недр
библиотеки, вместо ясной ошибки конфига при загрузке.

**Требование:** валидируй вход как можно раньше (схема/DTO), с понятным сообщением.
Не давай малформленному значению путешествовать до точки далёкого отказа.

### 3. Тихий «no match» хуже громкой ошибки
Сравнение/поиск, который на малформленном входе просто возвращает «ничего не нашлось»,
**прячет** баг: ни лога, ни сигнала. Такой путь молча отбрасывает корректный трафик.

**Случай:** `try_rx` возвращал `None` на несовпадении регистра → весь room-server
поток отбрасывался тихо.

**Требование:** на путях «не совпало → отбросить» убедись, что несовпадение не может
возникнуть из-за неинварианта (см. №1). Где уместно — логируй неожиданный дроп,
а не глотай молча.

### 4. Сверяй контракт API с реально установленной зависимостью, а не с памятью/доками
Перед тем как доверять вызову стороннего API, проверь **по факту установленной
версии**: сигнатуры, имена методов/полей, async-ность, имена полей payload
(интроспекция, чтение исходников зависимости). Версия в проде/локально может
отличаться от той, под которую писали.

**Случай:** код таргетил методы 2.x (`send_login_sync`, `send_msg_with_retry`,
`Event.is_error`), а резолвилась 1.x, где их нет — был бы `AttributeError` на железе.
Вскрыла это только сверка с установленной библиотекой.

**Требование:** места, где контракт ещё не подтверждён против реального устройства/
токенов, помечай явным маркером (`# verify`) и держи маркер до проверки на железе.
Контракт против установленной версии проверяй интроспекцией, а не по памяти.

---

## Чек-лист на границе с внешним кодом

- [ ] Внешние идентификаторы нормализованы к канонической форме на входе.
- [ ] Инварианты входа (длина/формат/диапазон) проверены в схеме, а не в IO.
- [ ] Ни одного пути, где малформленный вход тихо отбрасывается как «не совпало».
- [ ] Контракт стороннего API сверен с **установленной** версией (сигнатуры, поля,
      async); непроверенное на железе помечено `# verify`.
