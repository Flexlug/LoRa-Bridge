# Telegram: визуальная транслитерация (гомоглифы)

**Дата**: 2026-06-28
**Ветка**: worktree-tests-consolidation
**Слой**: `lora_bridge/transports/telegram/moderation/transliterate.py`

---

## 1. Контекст

`transliterate()` используется командой `set_transliter`: пользователь может включить
транслитерацию своих сообщений, чтобы ужать payload при отправке в эфир LoRa
(кириллица — 2 байта/символ в UTF-8, ASCII-латиница — 1 байт).

Текущая реализация — **фонетическая** транслитерация (таблица `str.maketrans`):
`привет → privet`, `ё → yo`, `щ → sch`. У неё два изъяна:

- многобуквенные замены (`щ→sch` = 4 байта против 2 у кириллицы) **увеличивают**
  payload — противоречит самой цели;
- латиница подбирается по звучанию, а не по виду — читающий не узнаёт исходное слово
  «по форме».

## 2. Цель

Сменить принцип подбора: для каждой кириллической буквы — **ближайший по начертанию**
ASCII-латинский аналог, чтобы вывод визуально напоминал оригинал. Payload — жёсткое
ограничение: замена не должна его увеличивать.

## 3. Правила маппинга (как принимали решение)

1. **Только тот же регистр.** Кросс-регистровые совпадения (строчная `н` ≈ заглавная `H`)
   запрещены — вывод не должен «прыгать» по регистру внутри строки.
2. **Без цифр.** `з→3`, `ч→4`, `б→6` исключены, хотя визуально точны.
3. **Только уверенные совпадения.** Слабые/средние аналоги (`э→e`, `ц→u`, `д→d`, `ж→x`,
   `й→u`, `щ→w`, `б→b`, `в→b`) отвергнуты — соответствующие буквы остаются кириллицей.
4. **Безнадёжные буквы остаются кириллицей.** Комбо из 2 ASCII = те же 2 байта, что и
   кириллица → экономии нет, поэтому комбинации не используются вовсе. (Правило «комбо
   только если уменьшает payload» в итоге не даёт ни одной замены — фиксируем это явно.)

Следствие — **регистровая асимметрия**: `В/Н/Т → B/H/T` (заглавные совпадают с латиницей),
но строчные `в/н/т` остаются кириллицей; `п→n`, но `П` остаётся. Это принято осознанно.

## 4. Таблица замен

**Строчные (15):**
`а→a  г→r  е→e  ё→e  и→u  к→k  м→m  о→o  п→n  р→p  с→c  у→y  х→x  ш→w  ь→b`

**Заглавные (14):**
`А→A  В→B  Е→E  Ё→E  Н→H  К→K  М→M  О→O  Р→P  С→C  Т→T  У→Y  Х→X  Ш→W`

**Без изменений** (passthrough): все прочие кириллические буквы
(`б в д ж з й л н т ц ч щ ъ ы э ю я` + соответствующие заглавные, кроме перечисленных),
латиница, цифры, пунктуация, пробелы, эмодзи.

Примеры:
```
привет             → npuвeт
Привет, как дела?  → Пpuвeт, kak дeлa?
объект             → oбъekт
ПРИВЕТ             → ПPИBET
Вася               → Bacя
hello 123!         → hello 123!   (без изменений)
```

## 5. Реализация

Та же форма, что сейчас — один `str.maketrans` + `text.translate(...)`. Меняется только
содержимое таблицы (см. §4). Сигнатура `transliterate(text: str) -> str` неизменна.
Символы вне таблицы `translate` оставляет как есть — passthrough получается бесплатно.

## 6. Тесты

Текущие параметризованные кейсы в `tests/test_telegram_moderation_transliterate.py`
проверяют **фонетику** (`privet`, `Privet`, `Vasya`, `obekt`) — их надо переписать под
новый вывод. Покрыть:

- маппинг строчных и заглавных (по представителю каждого класса);
- регистровую асимметрию (`В→B`, но `в` остаётся; `п→n`, но `П` остаётся) — это
  ключевой инвариант, легко сломать при правке таблицы;
- passthrough не-кириллицы и пустой строки;
- (опц.) проверка, что вывод по байтам ≤ входа на тексте только из маппящихся букв.
